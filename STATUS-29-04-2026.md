## Status por problema (ordem cronológica)

### 1. ✅ RESOLVIDO — `api-ms-win-core-path-l1-1-0.dll` faltando no Win 7

**Causa**: Python 3.5+ no Windows usa Universal C Runtime (UCRT). Win 7 sem
KB2999226 não tem. Win 10 antigos sem updates também não.

**Solução aplicada**:
- Build agora coleta DLLs do System32 e embute no .exe (5 DLLs: `ucrtbase.dll`,
  `vcruntime140.dll`, `vcruntime140_1.dll`, `msvcp140.dll`, `concrt140.dll`).
- Win 7 SP1 e Win 10 Pro agora abrem o exe sem erro de DLL.

**Arquivos modificados**:
- `C:\Sigatec coletor pro\installer\build.py` — função `_coletar_ucrt_dlls()` + flag `--add-binary` por DLL
- `C:\sigatec-coletor\sigatec-coletor\installer\sigatec_coletor.spec` — função `_coletar_ucrt_binaries()` no spec

### 2. ✅ RESOLVIDO — Build com Python 3.12 não roda em Win 7 (Python 3.10+ dropou Win 7)

### 3. ✅ RESOLVIDO — "API Key inválida" no Win 7

### 4. ✅ RESOLVIDO — `__pycache__` cacheava `config.py` com placeholder

**Causa**: o build importa `_xor_encode` de `coletor.config` no início, criando
`coletor/__pycache__/config.cpython-3X.pyc` com o placeholder original. Se
PyInstaller usar esse `.pyc` cacheado em vez de recompilar do `config.py`
modificado, o exe fica com placeholder e a chave decodifica vazia.

**Solução aplicada**:
- `build.py` agora apaga `coletor/__pycache__` DEPOIS de modificar `config.py`
  e ANTES de chamar PyInstaller, garantindo que o `.pyc` seja regenerado.

**Arquivos modificados**:
- `C:\Sigatec coletor pro\installer\build.py`
- `C:\sigatec-coletor\sigatec-coletor\installer\build.py`

### 5. ✅ RESOLVIDO — USB lê mas não extrai serial (Brother T-series)

**Sintoma**: USB detectada e lida (1039 bytes), parser extrai modelo + pagecount
mas não consegue serial. Sem serial, validação rejeita a leitura. Brother
**DCP-T520W** e **DCP-T730DW** afetadas.

**Log do bug**:
```
SetupAPI: 1 dispositivo(s) USBPRINT, 1 Brother
USB direto: resposta recebida (1039 bytes) mas não foi possível extrair serial+contador.
Dados parseados: {'modelo_raw': 'Brother DCP-T730DW:8CH-A47-001:Ver.1.09',
                  'modelo': 'DCP-T730DW', 'pagecount': 2302}
```

**Causa**: tank ink printers (T-series) não respondem a `@PJL INFO SERIALNUMBER`,
`BRSERIAL`, etc. O fallback existente (`obter_serial_usb_via_pnp`) usa
PowerShell `Get-PnpDeviceProperty`, que não existe em Win 7 (PowerShell 2.0).
Get-Printer e Get-PnpDeviceProperty só existem em PS 4.0+ (Win 8.1+).

**Soluções aplicadas (ainda não testadas)**:

1. **Fallback via registry** — função `_obter_serial_usb_via_registry()` em
   `usb_bidi.py` lê `HKLM\SYSTEM\CurrentControlSet\Enum\USB\VID_xxxx&PID_xxxx`
   e enumera subkeys, retornando o primeiro que não seja sintetizado pelo Windows.
   Funciona em Win 7 SP1+.

2. **Fallback do modelo_raw** — em `usb_reader.py`, se nem PJL nem registry
   acharam serial, extrai a parte central de `modelo_raw` (`Brother XXX:CODE:Ver.YYY`),
   ex: `8CH-A47-001`. Não é serial real, é código de fábrica, mas evita
   descartar a leitura.

**Arquivos modificados (não commitados ainda)**:
- `C:\Sigatec coletor pro\coletor\usb_bidi.py`
- `C:\Sigatec coletor pro\coletor\usb_reader.py`

### 6. ✅ RESOLVIDO — Tunnel WebSocket desconecta sempre

**Sintoma**: log enche de:
```
Tunnel desconectado (create_connection() got an unexpected keyword argument 'ping_interval').
Reconectando em 60s...
```

**Causa**: `websockets 13.1` (instalada no Python 3.8) tem um sync API
(`websockets.sync.client.connect()`) que **não aceita `ping_interval`** —
esse parâmetro só existe na API async/legacy. O código passava
`ping_interval=None` e a lib rejeitava.

**Solução aplicada**: remover `ping_interval=None` da chamada em
`coletor/tunnel.py`. O coletor já faz heartbeat manual via thread separada
(`TunnelHeartbeat`), então não precisa do ping automático da lib.

**Confirmacao de funcionamento**:
- Log mostra: `Tunnel conectado (installation_id=...)` em vez do
  `Tunnel desconectado` repetitivo.

**Arquivos modificados**:
- `C:\Sigatec coletor pro\coletor\tunnel.py`
